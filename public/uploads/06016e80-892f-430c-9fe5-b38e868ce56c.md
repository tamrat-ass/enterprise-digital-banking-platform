# Model Development System — MLOps Review & Corrected Workflow
### Ahadu Bank Digital Product Evaluation Platform

**Review type:** Data leakage / overfitting / validation / evaluation / deployment-readiness audit
**Verdict:** The original document is unusually candid about its own weaknesses, but two of the *most severe* issues aren't on its radar at all: **label leakage in the bootstrap formula** and **temporal/entity leakage in the train/test split**. Both quietly inflate every reported metric in the system. Everything below is organized as (1) what's actually broken, ranked by severity, then (2) the corrected pipeline.

---

## 1. Issues the Original Document Missed

### 1.1 🔴 Label leakage is worse than "circular dependency" — it's direct formula leakage

The document treats the bootstrap problem as a *feedback-loop* risk (today's predictions become tomorrow's training labels). That's real, but it undersells the first-generation problem: the bootstrap label is a **deterministic linear combination of features that are also fed into the model as inputs**.

```
label = f(txn_success_rate, active_user_rate, csat_score,
           complaint_resolution_rate, downtime_impact_score, api_error_rate)

features = [txn_success_rate, active_user_rate, csat_score,
            complaint_resolution_rate, downtime_impact_score, api_error_rate, ...]
```

When 6 of your 14 input features are literally the terms of the label formula, the model isn't learning "what makes a product perform well" — it's learning to invert a known linear equation. Reported accuracy/R² on the first generation of models is not a measure of predictive skill; it's a measure of how well linear regression can recover its own generating function (answer: almost perfectly). This means:

- Early "strong" metrics (82% accuracy, high R²) are **not evidence the system works** — they're an artifact.
- Any downstream decision ("model is ready, disable bootstrap") based on these numbers is unsound.
- The fix isn't just documentation (as the original doc proposes) — it requires either (a) a genuinely independent label source for at least the seed generation, or (b) explicitly holding out the bootstrap-derived features from the seed model and validating on a small hand-labeled/expert-reviewed set before trusting any metric.

### 1.2 🔴 Train/test split leaks across time and across product (data leakage)

```python
X_train_raw, X_test_raw, y_train, y_test = _safe_split(X_raw, y, test_size=0.2)
# stratified by tier, random_state=42 — no grouping, no time ordering
```

This dataset is a **panel**: multiple periods per product, with engineered features (`complaint_growth_rate`, MoM deltas) that are explicitly defined relative to the *previous period of the same product*. A random stratified split will:

- Put **Product A's March row in train** and **Product A's April row in test** — the model can partially memorize product-specific baselines (a given branch/product has a persistently high or low CSAT, transaction volume, etc.), so test performance overstates how well the model generalizes to a *new* product it has never seen.
- Occasionally split **adjacent periods of the same product** across train/test, where the test row's MoM feature was computed using data that appeared in a training row — a direct autocorrelation leak.
- Give no guarantee that training only sees "past" data relative to test — i.e., it doesn't simulate the real deployment condition (predicting the next period using only prior periods).

**Fix:** use a **grouped, time-respecting split** — see §2.3.

### 1.3 🟠 The test set is not actually held out — it's reused as a model-selection set

`select_best_model()` recomputes the *same* `random_state=42` split from the full table and evaluates every candidate model on it, then promotes a winner. That set has now been used for:
1. Hyperparameter selection (indirectly, since all models were tuned against a split derived from the same seed/data)
2. Final metric reporting
3. Model selection between algorithms

Once a test set influences *any* decision (which model to ship), it is no longer a valid estimate of production performance — it becomes an implicit validation set, and the "test" metrics reported to stakeholders are optimistically biased. There is currently no dataset in this pipeline that is touched exactly once. **Fix:** three-way split (train / validation / test), where test is only ever scored once per model generation, after the winner is already chosen.

### 1.4 🟠 `_cap_metrics()` masks the leakage symptom instead of fixing the cause

> "Cap metrics to prevent overestimation"

Artificially capping a metric because it looks "too good" is a workaround for §1.1/§1.2, not a fix. If a metric is unrealistically high, that's a diagnostic signal pointing at leakage — suppressing the number instead of investigating it means the team loses the one signal that would have caught this. **This function should be deleted**, not tuned; once leakage is fixed, metrics will land in a believable range on their own.

### 1.5 🟠 Non-reproducible training runs

```python
return X + np.random.normal(0, 0.05, X.shape)   # no seed
```

Every other stochastic component in the pipeline is seeded (`random_state=42`), but the "regularization noise" is not. Two training runs on identical data with identical hyperparameters will produce different models and different metrics, which breaks reproducibility, complicates debugging, and makes CI-based regression testing of model quality impossible. Also worth reconsidering on merits: L2 regularization is already being tuned via `C`/`alpha` in `GridSearchCV` — adding unseeded Gaussian noise on top is a second, uncontrolled regularizer whose interaction with the first is not understood.

### 1.6 🟠 Classifier is trained but frequently overridden — its metrics don't reflect deployed behavior

The "safety check" silently replaces the classifier's tier with a score-derived tier whenever they disagree. This is reasonable as a production guardrail, but it means the F1/accuracy reported for the classifier at training time **does not reflect what tier the user actually sees**, since disagreements (which are exactly the hard cases) are overwritten. Model-selection-by-F1 (`select-best-model`) is therefore optimizing a metric that's disconnected from the deployed decision. Recommend logging an **"effective tier accuracy"** metric computed *after* the override, so the ensemble comparison reflects real-world behavior.

### 1.7 🟡 No baseline / naive model for comparison

There's no `DummyClassifier` (majority class) or `DummyRegressor` (mean) baseline recorded anywhere. Without one, "82% accuracy" or "R² = 0.75" can't be judged — if the tier distribution is 70% MEDIUM, a model that always predicts MEDIUM gets 70% "accuracy" for free. Every training run should log the naive baseline alongside the model's score.

### 1.8 🟡 Similarity (KNN) model has no evaluation at all

`train_similarity()` fits on 100% of the data with no held-out check — there's no way to know if the "similar products" it returns are actually meaningful versus noise, especially at low `n_neighbors` on small datasets.

### 1.9 🟡 Confidence scores are not real confidence

```python
confidence = 0.85   # hardcoded, always, regardless of input
```

This is presented to users as if it were calibrated uncertainty, but it's a constant. This is a trust/compliance risk in a banking context — decisions may be made based on a "confidence: 0.85" that means nothing. Should be replaced with a real interval (quantile regression, bootstrap ensembles, or at minimum distance-to-training-distribution).

### 1.10 🟡 Deployment readiness gaps not on the checklist

The "Production Readiness Checklist" covers modeling hygiene well but is missing standard MLOps deployment mechanics entirely:
- No CI/CD gate that blocks promotion if metrics regress vs. the currently-active model
- No shadow/canary rollout before full promotion (listed as "nice to have" — should be **required**, since this is a bank scoring product performance, likely feeding decisions)
- No automated rollback trigger tied to monitoring (rollback is only "documented," not automated)
- No point-in-time feature store guarantee (needed to prevent future leakage as more engineers touch this)
- No SLA/latency or load testing on `/ml/predict`

## 2. Corrected, Production-Grade Workflow

### 2.1 Label strategy — break the circular dependency for real

```
Generation 0 (seed):
  Bootstrap formula → labels_v0
  Train model_v0 on labels_v0
  Tag model_v0 as "seed" — NEVER auto-promoted to serve traffic
  Validate against a small, independently-sourced sample:
    - Domain-expert-scored products (target: 30-50 products, reviewed by product/ops team)
    - Compute agreement (Spearman correlation on score, tier confusion matrix)
    - Only promote model_v0 to serving if agreement clears an approved threshold

Generation 1+ (once real Score history + expert-reviewed labels accumulate):
  labels = blend of (a) expert-reviewed scores where available,
                     (b) prior period's *served* prediction, weighted down over time,
                     (c) bootstrap only as last-resort fallback, flagged in metadata
  Every training run logs: % of labels from each source
  Alert if % bootstrap-derived labels > threshold (e.g. 20%) — surfaces silent bootstrap creep
```

Additionally: **exclude bootstrap-formula input features from being simultaneously used as model features when training on bootstrap-derived labels**, or explicitly disclose that early-generation metrics reflect formula recovery, not real predictive skill — don't let them appear in the same dashboard as later "real" metrics without a generation marker.

### 2.2 Feature engineering — enforce point-in-time correctness

```python
# app/core/features.py — single source of truth (as the original doc proposed),
# PLUS point-in-time validation
FEATURE_REGISTRY = {
    "active_user_rate": {"db_column": "active_user_rate", "type": "float",
                          "min": 0.0, "max": 1.0, "lag_periods": 0},
    "complaint_growth_rate": {"db_column": "complaint_growth_rate", "type": "float",
                               "lag_periods": 1,   # explicitly depends on t-1
                               "description": "MoM change vs prior period"},
    # ...
}

def validate_no_future_leakage(df: pd.DataFrame, as_of_date):
    """Assert no row's features were computed using data dated after as_of_date."""
    assert (df["period_date"] <= as_of_date).all()
    assert (df["feature_computed_at"] <= as_of_date).all()  # add this column
```

Every `ProcessedFeatures` row should carry a `feature_computed_at` timestamp, separate from `period_date`, so leakage audits are mechanically checkable rather than assumed correct.

### 2.3 Data split — grouped + time-respecting, three-way

```python
def production_split(df: pd.DataFrame, test_periods=2, val_periods=2):
    """
    1. Sort by period_date.
    2. Hold out the MOST RECENT `test_periods` months as TEST — simulates real
       deployment (train on past, predict future). Never touched until final eval.
    3. Hold out the `val_periods` months before that as VALIDATION — used for
       hyperparameter tuning and model selection.
    4. Everything earlier is TRAIN.
    5. Within any period, do NOT allow a product to be split across sets in a way
       that creates train/test adjacency for the same product (naturally satisfied
       here since splitting is by whole time periods, not random rows).
    """
    periods = sorted(df["period_date"].unique())
    test_cutoff = periods[-test_periods]
    val_cutoff = periods[-(test_periods + val_periods)]

    train = df[df.period_date < val_cutoff]
    val   = df[(df.period_date >= val_cutoff) & (df.period_date < test_cutoff)]
    test  = df[df.period_date >= test_cutoff]
    return train, val, test
```

If the business genuinely needs to generalize across *new, unseen products* rather than *future periods of known products*, add `GroupKFold`/`GroupShuffleSplit` by `product_id` on top of this — but for a monitoring/scoring product where the same set of products recur monthly, time-based splitting is the split that matches how the model will actually be used.

Cross-validation during hyperparameter tuning should use **`TimeSeriesSplit`** (or blocked/grouped CV) instead of plain `StratifiedKFold`, for the same reason.

### 2.4 Training pipeline (corrected)

```
1. LOAD DATA  → train / val / test via production_split()
2. PREPARE FEATURES → single FEATURE_REGISTRY source, schema-validated
3. LABELS → generation-aware label strategy (§2.1), log source mix
4. SCALE
     - StandardScaler fit on TRAIN ONLY (unchanged — this part was already correct)
     - Remove the unseeded Gaussian noise step, OR seed it explicitly and treat
       it as a documented, tunable hyperparameter (prefer removing it: rely on
       C/alpha regularization + more CV folds instead)
5. HYPERPARAMETER TUNING
     - RandomizedSearchCV or Optuna instead of fixed grids (as original doc proposed)
     - CV = TimeSeriesSplit / blocked CV on TRAIN only
     - Selection metric evaluated on VALIDATION, not test
6. TRAIN final model on TRAIN (+ val, optionally, once hyperparams are locked)
7. EVALUATE ONCE on TEST (untouched until now)
     - Full metric suite (see §2.5) — no capping
     - Compare against DummyClassifier/DummyRegressor baseline
     - Compare against currently-active production model (regression gate)
8. BUNDLE ARTIFACTS
     bundle = {model, scaler, label_encoder, feature_names, feature_registry_version,
               training_date, git_commit, label_source_mix, val_metrics, test_metrics}
     joblib.dump(bundle, f"{model_type}_{version}.pkl")
9. PROMOTION GATE (new)
     - Auto-block promotion if test metrics regress vs. active model beyond tolerance
     - Require shadow-mode run (predict alongside active model, log both, don't serve)
       for N days before promotion to full traffic
10. REGISTER + PROMOTE (as before, but promotion is gated, not automatic)
```

### 2.5 Evaluation — full suite, computed once, never capped

Log to `ModelRegistry` per training run:

| Category | Metrics |
|---|---|
| Classification | Accuracy, weighted F1, per-class precision/recall/F1, confusion matrix, ROC-AUC (one-vs-rest), log loss |
| Regression | R², MAE, RMSE, residual plot data, calibration (predicted vs. actual by decile) |
| Baseline comparison | Same metrics for Dummy baseline, delta vs. model |
| Stability | CV metric mean ± std across folds (flags high-variance/overfit models) |
| Fairness/segment check | Metrics broken out by product category/segment, if available — catches a model that's great on average but bad for one tier |

### 2.6 Deployment readiness additions

- **CI gate:** training pipeline runs in CI on every PR touching `ml_service.py` / `feature_engineering.py`; blocks merge if a fixed regression-test dataset's metrics drop beyond tolerance.
- **Shadow deployment:** new model version predicts in parallel with the active model for a fixed window; predictions logged but not served. Compare agreement before promoting.
- **Canary rollout:** 5% → 25% → 100% traffic, with automatic rollback if error rate or prediction-distribution drift trips a threshold.
- **Automated rollback:** tie `/ml/check-drift` and a new `/ml/check-prediction-drift` (distribution of *outputs*, not just input features) to an alert that can trigger `_promote_to_latest()` reverting to the last known-good version.
- **Load/latency testing** on `/ml/predict` before go-live, since this feeds a production banking dashboard.
- **Point-in-time feature store discipline** enforced via `validate_no_future_leakage()` in CI, not just code review.

## 3. Prioritized Action Plan

**Before any model is trusted for real decisions:**
1. Fix the train/test split to be time-respecting (§2.3) — every metric in the system is currently unreliable without this.
2. Stop treating bootstrap-generation metrics as evidence of model quality (§2.1); validate against an independent expert-labeled sample.
3. Delete `_cap_metrics()`; let real numbers surface problems instead of hiding them (§1.4).
4. Move to a train/val/test three-way split so "best model selection" doesn't contaminate the reported test metric (§1.3).
5. Seed (or remove) the training noise step for reproducibility (§1.5).

**Before production rollout:**
6. Add baseline model comparison to every training run (§1.7).
7. Add shadow-mode + canary rollout + automated rollback (§2.6).
8. Replace hardcoded `confidence = 0.85` with a real uncertainty estimate (§1.9) — non-negotiable in a banking context.
9. Centralize `FEATURE_REGISTRY`, bundle artifacts together (both already correctly identified in the original doc).

**Next sprint:**
10. RandomizedSearchCV/Optuna instead of fixed grids; TimeSeriesSplit for CV.
11. Full metric suite (confusion matrices, per-class metrics, ROC-AUC) stored per model.
12. Prediction-distribution drift monitoring, not just feature drift.
13. Evaluate the KNN similarity model with a held-out check.

## 4. What Was Already Right

Worth preserving as-is: fitting the `StandardScaler` on training data only, batch-querying labels to avoid N+1s, chunked loading for large uploads, model versioning with timestamped artifacts, and RBAC on training endpoints. The instinct to keep a "strengths/weaknesses" self-audit in the docs at all is good practice — the corrections above should be folded into that same document going forward so drift-in-documentation doesn't become its own problem.
